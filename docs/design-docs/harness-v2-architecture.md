# Harness v2 — 듀얼 에이전트 교차 검증 아키텍처

> 기존 Planner-Generator-Evaluator 단일 모델 패턴에서
> Claude Code + Codex 교차 검증 패턴으로 전환한다.

## 1. 문제 정의

### 기존 하네스 (v1)의 한계

1. **자기 과대 평가**: 같은 모델(Claude)이 구현과 평가를 모두 수행 → 자기 편향
2. **컨텍스트 비대화**: 라운드가 누적될수록 blackboard, journal이 비대해져 집중력 저하
3. **프로세스 품질 미측정**: 결과물만 평가하고, 결과물에 도달하는 과정의 품질은 무시
4. **검증 수단 부재**: 코드 리뷰만 존재, 테스트/스크린샷/grader 등 자동화 검증 없음

### Claude Code 내부 하네스에서 배운 것

- **High-signal 컨텍스트**: 길게 주는 것이 아니라 필요한 정보만 정밀하게 제공
- **Grader 내장**: 평가자에게 채점 기준표(rubric)를 주고 항목별 점수화
- **테스트 기반 검증**: 코드 리뷰 외에 실제 테스트 실행 결과를 평가 입력으로 사용
- **스크린샷 검증**: UI 변경은 시각적 결과물로 검증
- **반복 수렴**: 10+ 라운드 반복으로 품질 극대화 — 그러나 각 라운드의 입력은 lean하게

## 2. 새 아키텍처

```
┌────────────────────────────────────────────────┐
│                    Human                        │
│         (의사결정, 모호성 해소, 최종 승인)        │
└──────────┬─────────────────────┬───────────────┘
           │                     │
    ┌──────▼──────┐       ┌──────▼──────┐
    │ Claude Code │◄─────►│   Codex     │
    │  (Primary)  │       │ (Verifier)  │
    └──────┬──────┘       └──────┬──────┘
           │                     │
           ▼                     ▼
    ┌─────────────────────────────────────┐
    │          Shared Artifacts           │
    │  docs/ ← SKILLS.md ← SECURITY.md   │
    │  context_harness/ ← harness-logs/   │
    └─────────────────────────────────────┘
```

### 2.1 역할 분담

| 단계 | Claude Code | Codex | 산출물 |
|------|------------|-------|--------|
| **공동 계획** | 전체 방향 + 제약 정의 | 세부 설계 + 의존성 분석 | `docs/exec-plans/` |
| **설계** | 검토 + 승인 | 초안 작성 | `docs/design-docs/` |
| **구현** | 코드 작성 | — | git commit |
| **1차 검증** | — | 코드 리뷰 + 수정 패치 | review report |
| **2차 검증** | 교차 확인 + 머지 판단 | — | merge decision |
| **회고** | SKILLS.md 업데이트 | 독립 회고 | `docs/harness-logs/` |

### 2.2 교차 검증 원칙

```
구현한 에이전트 ≠ 1차 검증 에이전트 ≠ 2차 검증 에이전트

Claude Code 구현 → Codex 1차 검증 → Claude Code 2차 검증 (교차)
Codex 설계 → Claude Code 검토 → 합의 후 진행
```

- 서로 다른 모델의 관점을 활용하여 blind spot 최소화
- 1차 검증자가 놓친 것을 2차 검증자가 잡는 구조
- 양쪽 모두 동의해야 머지

## 3. 컨텍스트 관리 — High Signal 원칙

### 3.1 에이전트에게 주는 것

| 구분 | 포함 | 제외 |
|------|------|------|
| 평가 입력 | 변경된 파일 diff, 관련 acceptance criteria, 빌드/테스트 결과 | 전체 코드베이스, 과거 라운드 전체 로그 |
| 설계 입력 | PRD 관련 섹션, 기존 아키텍처 요약, 제약 조건 | 전체 PRD, 전체 하네스 히스토리 |
| 구현 입력 | exec-plan의 해당 태스크, 관련 파일, SECURITY.md | 전체 exec-plan, 다른 태스크 상세 |

### 3.2 컨텍스트 압축 규칙

- 각 라운드 시작 시 이전 라운드 결과를 **3줄 요약**으로 압축
- blackboard는 **현재 활성 결정사항만** 유지 (아카이브 분리)
- 평가 리포트는 `docs/harness-logs/`에 누적, 활성 컨텍스트에서는 최신 1개만

## 4. 검증 체계

### 4.1 자동 검증 (하네스 내장)

| 검증 유형 | 도구 | 통과 기준 |
|----------|------|----------|
| 타입 체크 | `npx tsc --noEmit` | 에러 0 |
| 빌드 | `npx next build` | 성공 |
| 린트 | `npx eslint .` | 에러 0 (경고 허용) |
| 보안 | SECURITY.md 체크리스트 자동화 | 전 항목 통과 |
| 접근성 | WCAG 명암비 계산 | 4.5:1+ |

### 4.2 코드 리뷰 (교차 에이전트)

- 구현 에이전트가 아닌 다른 에이전트가 수행
- Grader rubric 기반 항목별 점수화:
  - 보안 (0-10), 기능 정확성 (0-10), 접근성 (0-10), 성능 (0-10), 코드 품질 (0-10)
- 프로세스 품질도 측정:
  - 커밋 메시지 명확성, 변경 범위 적절성, 테스트 커버리지

### 4.3 시각적 검증

- UI 변경 시 스크린샷 첨부 또는 dev 서버에서 실제 확인
- OG 이미지는 실제 렌더링 결과 확인

## 5. 하네스 로그 구조

```
docs/harness-logs/
├── round-028.md          # 라운드별 결과
├── round-029.md
├── ...
├── quality-trend.md      # 품질 추이 차트
└── retrospective-001.md  # 주기적 회고
```

각 라운드 로그 형식:
```markdown
# Round {N} — {date}
## 입력: {어떤 변경이 평가 대상이었는가}
## 검증자: {누가 검증했는가}
## 결과: {PASS|CONDITIONAL_PASS|BLOCK}
## Findings: {finding 목록}
## 프로세스 점수: {rubric 점수}
## 조치: {무엇을 수정했는가}
## 교훈: {SKILLS.md에 기록할 것이 있는가}
```

## 6. 실행 병렬화

독립적인 작업은 병렬로 실행:
- Claude Code가 Epic A 구현하는 동안 Codex가 Epic B 설계
- 빌드 검증과 코드 리뷰를 동시 실행
- 검증 통과된 작업부터 순차 머지

의존성이 있는 작업은 순차 실행:
- DB 스키마 변경 → 이를 사용하는 API → 이를 호출하는 UI
