## Decision

**ACCEPTED all 11 bypasses + 3 residual drift items.** Implementation strategy per Codex R4 convergence:

### (a) Tamper-evident primitives (trust model)

- **Lock event log** — per round: `context_harness/operator/locks/<round>.events.jsonl`. Appended on `created` / `amended` / `closed`. Schema: `{ts, action, lock_sha256, base_commit, amendment_file|null}`. Checker at `gates`/`close` recomputes current lock sha and blocks if it doesn't match the last event. `close` is the only action allowed to advance the sha (records post-close sha per Codex's 6th must-fix).
- **Gate evidence hashed at close** — `close` computes `sha256(gate_evidence.json)` and records `gate_evidence_sha256` in lock. Later edits detectable.
- **Codex identity presence checks** — meeting frontmatter adds optional `codex_session_id` (regex `[A-Za-z0-9._:-]{8,}` per Codex R4) and/or `codex_transcript` (file path must exist). Meetings with `codex` in participants require at least one. Presence only — content trusted per honest-agent model.

### (b) Amendment bypass closure

- Checker scans `round_dir.glob("*.amendment.*")` — every on-disk amendment must be in `lock.amendments[]` exactly once
- Every amendment requires lock metadata: `file`, `target`, `sha256` (starts with `sha256:`), `supersedes`, `meeting`
- For `.md` amendments: frontmatter `target`/`supersedes`/`meeting` must exist AND match lock entry
- Referenced meeting must have same `round` as lock, `status: decided`, `stage: operator_amendment` (or relevant contract stage), and body contains `## Amendment Detail` with amendment filename
- `.txt` amendments: lock metadata required; body format (`+path` / `-path` lines) validated

### (c) Gate evidence schema strengthening

Per-gate required fields:
- `gate1`: `status`, `command`, `exit_code`, `test_count`, `log` (path + sha256)
- `gate2`: `status`, `reports` (array of 3 with path+sha)
- `gate3`: `status`, `cross_agreement_note` (string) OR `summary` path+sha
- `gate4`: `status`, `metrics_source` path+sha, `remediation_cycles`, `blocker_recurrence`

Checker verifies: fields present, referenced files exist, SHAs match.

### (d) stages_completed demotion

Demoted to informational. Checker validates ID membership only, does NOT use for gate decisions.

### (e) cmd_lock pre-existing amendment failure

`cmd_lock` refuses if any `*.amendment.*` file exists in round_dir. Legitimate restart = delete round dir + lock and re-create.

### (f) Residual doc drift

- REGULATION line 120 changed: "effective state computed on read, not regenerated"
- PROCESS_AUDIT line 19: "stage IDs + pointers only; role matrix full consistency NOT enforced in v5.x"
- CHANGELOG v5.2 13/14 note: identified failing case (`deep/**` matching `deep` alone — documented as accepted edge case since git doesn't emit bare directory names)

## Challenge Section

### Objection (recorded)
Codex's "lock tampering bypasses everything" is technically correct but the adversarial threat model is out of scope for v5.3. I pushed back by proposing the honest-agent+tamper-evident scope; Codex agreed. No unresolved objection — this IS the architectural decision the meeting was called for.

### Risk
Tamper-evident via append-only JSONL files is defeated if an operator deletes the events file. Mitigation: git commits the events file; any deletion shows up in `git log`. Residual risk accepted: a malicious operator who's also willing to rewrite git history can still win. That's the honest-agent scope boundary.

### Rejected alternative
Implementing cryptographic signatures for Codex responses. Rejected because: (a) Codex CLI doesn't sign output; (b) introducing keypair management adds significant infrastructure; (c) the user explicitly chose this single-machine single-user setup. Deferred explicitly with a clear trigger: if we ever move multi-machine, reopen this peer decision.

### Uncertainty deferred
`stages_completed` demotion is the safer v5.3 choice, but if operators start relying on it informally we'll need strict validation later. Flag in first-real-round retro.

## Decision
PROCEED with v5.3 implementation. Core changes: lock event log, amendment disk-scan + metadata validation, gate evidence schema with hashes, Codex identity presence, stages_completed demotion, cmd_lock pre-existing amendment refusal, residual doc fixes. Commit + push after 0 blockers.
