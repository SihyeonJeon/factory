## §3. Contract & Lock

### Per-round directory

```
operator/contracts/<round_id>/
├── spec.md                      ← base input spec. IMMUTABLE after lock.
├── spec.amendment.1.md          ← amendments only as new files
├── spec.amendment.N.md
├── (spec.effective.md, acceptance.effective.md, eval_protocol.effective.md, file_whitelist.effective.txt — consumers compute on read from base+amendments; NOT regenerated by checker)
├── file_whitelist.txt           ← writable paths for this round. IMMUTABLE after lock.
├── file_whitelist.amendment.N.txt  ← whitelist amendments (additive or subtractive)
├── convention_version.txt       ← active coding-conventions path + SHA at lock time
├── lint_config.txt              ← active lint-config path + SHA at lock time
├── acceptance.md                ← success criteria. IMMUTABLE after lock.
├── acceptance.amendment.N.md    ← acceptance amendments
├── eval_protocol.md             ← Codex-authored capture protocol. IMMUTABLE after lock.
└── eval_protocol.amendment.N.md ← eval protocol amendments
```

### Amendment artifact format

All amendments share a common structure:
- Filename: `<base_filename>.amendment.<N>.<ext>` where N is a monotonically increasing integer per target
- YAML frontmatter (required, validated by checker):
  ```yaml
  ---
  target: <base_filename>
  supersedes: [<section-id or line-range>]
  meeting: <path-to-decided-meeting>
  ---
  ```
- Recorded in lock's `amendments[]` array with `{file, target, sha256, supersedes, meeting}`
- Consumers compute effective state by reading base + amendments in order. The checker **validates** amendments (file exists + sha matches lock + frontmatter present + meeting exists and `status: decided` + target ∈ base files) but does NOT regenerate `.effective.*` artifacts — consumers (e.g., operators running checks) compute effective state on read.

### Tamper-evident lock event log (v5.3+)

Per round: `operator/locks/<round_id>.events.jsonl` (append-only JSONL). Each event:

```json
{"ts":"ISO","action":"created|amended|closed","lock_sha256":"sha256:...","base_commit":"...","amendment_file":null}
```

Checker at `gates`/`close`/`amend` computes the current `lock_sha256` and compares to the last event's `lock_sha256`. Mismatch = Gate 5 blocker (tampering suspected). SHA-advancing actions:
- `lock` → emits `created` event
- `amend` → validates amendment, writes new lock, emits `amended` event with post-amend sha. Rolls back (restores pre-amend lock text) if post-write validation fails; event is NOT emitted in the rollback path.
- `close` → validates gates + gate evidence, writes closed lock, records `gate_evidence_sha256`, emits `closed` event with post-close sha.

Hand-editing status, amendments, or hashes in the lock without an evented command causes the next `gates` run to fail with a sha mismatch. Deleting the events file is detectable via `git log`. Accepted scope limit: this is honest-agent tamper-evidence, not malicious-fabrication resistance.

### Post-close gate evidence revalidation (v5.4)

When `lock.status == "closed"` and `lock.gate_evidence_sha256` is set, every `gates` run recomputes the live `gate_evidence.json` sha and compares. Mismatch = Gate 5 blocker. This closes the v5.3 regression where post-close edits were undetected (confirmed empirically during round_deepsight_r1 testing).

### Gate evidence schema (v5.3)

`operator/contracts/<round_id>/gate_evidence.json` — required by `close`. Schema:

```json
{
  "gate1": {"status":"pass","command":"xcodebuild test","exit_code":0,"test_count":140,"log":{"path":"...","sha256":"sha256:..."}},
  "gate2": {"status":"pass","reports":[{"path":"...","sha256":"..."},{"path":"...","sha256":"..."},{"path":"...","sha256":"..."}]},
  "gate3": {"status":"pass","cross_agreement_note":"..."},
  "gate4": {"status":"pass","metrics_source":{"path":"...","sha256":"..."},"remediation_cycles":2,"blocker_recurrence":0}
}
```

Checker validates: all fields present per-gate; any referenced path exists; SHAs match live files. `close` records `sha256(gate_evidence.json)` into `lock.gate_evidence_sha256` and appends a `closed` event with the post-close lock sha.

### Pointer file format (`lint_config.txt`, `convention_version.txt`)

Each is a 2-line UTF-8 file:

```
<repo-relative path to the active resource>
sha256:<64-hex>
```

Example for `lint_config.txt`:
```
context_harness/operator/lint_config.toml
sha256:a1b2c3...e7f8
```

Malformed format (missing lines, wrong prefix) = Gate 5 blocker.

### Lock file schema

`operator/locks/<round_id>.lock` is a single JSON object:

```json
{
  "round_id": "round_deepsight_r1",
  "schema_version": 1,
  "started_at": "2026-04-19T12:00:00Z",
  "status": "active",
  "operators": {
    "claude_code": { "session": "this-session", "role_default": "implementer" },
    "codex": { "session": "019db43d-...", "role_default": "designer_reviewer" }
  },
  "hashes": {
    "spec.md": "sha256:...",
    "file_whitelist.txt": "sha256:...",
    "convention_version.txt": "sha256:...",
    "lint_config.txt": "sha256:...",
    "acceptance.md": "sha256:...",
    "eval_protocol.md": "sha256:..."
  },
  "amendments": [
    { "file": "spec.amendment.1.md", "sha256": "...", "supersedes": ["spec.md#section-3"], "meeting": "meetings/2026-04-20_amend1.md" }
  ],
  "stages_completed": ["overall_planning", "detailed_design"],
  "closed_at": null
}
```

`status` values: `active` | `paused` | `escalated` | `closed` | `aborted`.

### Immutability rules

- Base contract files (`spec.md`, `file_whitelist.txt`, `convention_version.txt`, `lint_config.txt`, `acceptance.md`, `eval_protocol.md`) are **NEVER edited** after lock creation.
- Direct diff to any base file while `status == active` = hard stop. Checker refuses to advance stages.
- Changes require: (a) amendment meeting → (b) new `<base_filename>.amendment.<N>.<ext>` with frontmatter `target`, `supersedes`, `meeting` → (c) lock's `amendments[]` appended with `{file, target, sha256, supersedes, meeting}` → (d) consumers compute effective state on read; no `.effective.*` regeneration by checker.
- **Whitelist changes** (both expansion AND shrinking) go via `file_whitelist.amendment.N.txt` amendment. No exception. "Shrink without meeting" is forbidden — mutating the locked hash, regardless of direction, breaks the immutability invariant.
- `status` transitions: `close <round>` subcommand of `harness/check_operator_round.py` transitions to `closed` only if (a) Gate 5 structural integrity passes AND (b) `contracts/<round>/gate_evidence.json` is present with `gate1..gate4` all marked `pass`. The checker enforces Gate 5 directly and refuses close if the `gate_evidence.json` is missing, malformed, or any gate is not `pass`. Operators may transition to `paused` or `escalated` via meeting; `aborted` only via human. Operators may NOT hand-edit `status` in the lock file.
- `lint_config.txt` and `convention_version.txt` contain `<path>\n<sha256>`. Checker at Gate 5 recomputes the live SHA of the referenced path and compares — mismatch = blocker.

## §4. Challenge Section (anti-hallucination tripwire)

Every meeting whose purpose is a **decision** (normative, design-affecting, or contract-changing) must include a Challenge Section with at least ONE of:

- Concrete objection with reasoning
- Named risk with mitigation or acceptance
- Rejected alternative with why-not
- Explicit uncertainty that was consciously deferred

Meetings that are **purely factual** (verification, status check, evidence pointer) must state: **"No normative decision; verified facts only"** and list the evidence references.

Forbidden: fake dissent (reviewer nitpicking to satisfy the rule). If the reviewer genuinely has no objection, the meeting is factual by definition — state that.

Missing Challenge Section on a decision meeting = Gate 5 blocker.

## §5. Gate Structure

Gates 1–4 from v4 retained with enforcement clarified; Gate 5 added.

| Gate | Name | Check | Blocker |
|------|------|-------|---------|
| 1 | Build & Test | `xcodegen generate && xcodebuild test` passes, test count ≥ baseline | Any failure |
| 2 | 3-Evaluator Review | red_team + hig_guardian + visual_qa reports produced | Any evaluator missing or empty |
| 3 | Cross-Agreement | 2+ evaluators agreeing on a file → required fix | Unresolved 2+ agreement |
| 4 | Process Metrics | remediation cycles ≤ 2, BLOCKER recurrence = 0, brief accuracy ≥ 90%, SKILLS miss ≤ 5% | Any metric out of range |
| 5 | Process Integrity | see §5.1 below | see §5.1 blockers |

### §5.1 Gate 5 — Process Integrity

Blockers (round cannot close):
- Any commit in this round not traceable to a contract lock
- Any edit to a base contract file (§3 immutability)
- Any decision meeting without a Challenge Section
- Any new operator/* file not registered in `FILE_INDEX.md`
- Lock hash mismatch for any base contract file
- Lint checker missing, failing, or unrunnable (`harness/check_operator_round.py` exit != 0)
- Operator-layer drift audit reports contradictions (see §7)
- **v5.7: any commit where Claude Code Operator directly authored a Swift source change.** Swift implementation must be dispatched to Codex (detectable via git log + commit-author + Codex-transcript cross-check). Forward-looking from R5.

Advisories (noted in retro, do not block):
- No new SKILLS.md entry — acceptable if retro explicitly classifies "no novel pattern"
- FILE_INDEX line-count warning (between 250 and 300 lines)
- SESSION_RESUME.md staleness > 48h

## §6. Deadlock Escalation Ladder

When two operators cannot converge in a meeting:

1. **Reframe meeting** — each operator restates the OTHER'S position in their own words. Names the real decision axis. Frequently resolves miscommunication-based disputes.
2. **Default-reversible rule** — if the dispute is low-impact and reversible, pick the option with smaller blast radius, easier rollback, or better auditability. Record both positions; implementer proceeds. No human, no Gemini.
3. **Gemini advisory** — for **bounded technical disputes only** (lint rule choice, file layout, test strategy, refactor approach). Gemini provides ANALYSIS, not VERDICT. Claude/Codex still make the decision and record whether they accepted or rejected the advice.
4. **Human escalation** — required for: product-impact, user-visible design direction, security/privacy policy, architecture change, irreversible migration, or unresolved operator deadlock after Gemini advisory.

Timebox: 2 written counter-rounds OR 24 hours — whichever first — before moving to the next tier.

Lock `status` transitions to `escalated` at Tier 4. Must record: both positions, tier path taken, what was asked of the human.

## §7. Operator-Layer Drift Audit

Runs at Gate 5 and after every operator-doc amendment. Implemented in `harness/check_operator_round.py` (section `audit_operator_layer`).

**Checks currently enforced by the checker** (authoritative — if it's not in this list, don't assume it's checked):
1. Stage IDs consistent across `lint_config.toml`, `STAGE_CONTRACT.md`, and `MEETING_PROTOCOL.md`
2. Every path-like reference in operator docs resolves (repo-root OR `operator/` shorthand inside operator docs); unresolved = blocker unless in configured allowlist
3. Every operator/*.{md,toml,yaml,yml} file is indexed in FILE_INDEX.md (except meetings, proposals, retros, contracts, locks, CHANGELOG, FILE_INDEX itself)
4. Line-count caps: AGENTS.md ≤80, .claude/CLAUDE.md ≤80, OPERATOR.md ≤120, FILE_INDEX.md ≤250 (warn at 225)
5. Loader pointers: `.claude/CLAUDE.md` and `AGENTS.md` reference `context_harness/operator/OPERATOR.md`
6. Legacy SUPERSEDED markers on docs listed in `lint_config.superseded_docs` (first 10 lines must contain SUPERSEDED or ACTIVE)
7. SESSION_RESUME.md freshness: ≤48h = pass, 48h-7d = advisory, >7d = blocker
8. Meeting files: required frontmatter keys, allowed stage/status values, Challenge Section content rule (normative stages require ≥1 keyword; factual meetings require "No normative decision" phrase + `Evidence:` section with ≥1 path/command reference)
9. `lock.stages_completed[]` entries are valid stage_ids from lint config

Any contradiction on (1)-(9) → Gate 5 blocker. Resolution path: amendment meeting.

**Explicitly NOT enforced in v5.2** (do not cite as audited):
- Full role matrix consistency (agent/role/stage cross-check against STAGE_CONTRACT table) — structural stage_id validation only.
- Process-log event count matching meeting decision count (advisory in PROCESS_AUDIT, not blocked).
- Effective amendment file regeneration (consumers compute on read).
- Gate 1-4 content (build/test/evaluator reports) — these come from `gate_evidence.json`, not this audit.

If operators need any of the "not enforced" items, open an amendment meeting to either implement or keep REGULATION honest.

## §12. Multi-Axis Evaluation (v5.7)

Gate 2 MUST include evidence across 5 axes for every implementation round. Missing any axis = Gate 5 advisory (not blocker) with retro note; recurring misses become blocker in v5.8.

1. **Code axis** — unit/integration tests; forbidden-pattern grep; static analysis. `xcodebuild test` pass.
2. **Runtime functional axis** — simulator boot + install + launch; primary user flow exercised (or explicit capture exception); screenshot or XCUITest log.
3. **UI/UX fidelity axis** — screenshot compared against deepsight mock (SHA-referenced); Codex written diff-style observation on alignment, color, typography, spacing, hierarchy.
4. **Navigation + info consistency axis** — tab/surface reachability verified; all user-facing strings localized; no English leaks; info hierarchy readable at system text sizes.
5. **Process-context axis** — meeting trail intact; Challenge Section with real objections; amendment flow followed; operator did not shortcut (e.g., direct Swift edit, unlogged deviations).

## §13. Vibe-Coding Regulation (v5.7)

Reference: `docs/design-docs/vibe-coding-limits-2026.md`. Every implementation dispatch to Codex MUST:

- Cite the vibe-coding-limits-2026 section(s) being guarded against in the dispatch prompt.
- Require the returned Swift to include at least a comment block `// vibe-limit-checked: <item-ids>` for the anti-patterns relevant to the touched surface.
- Be reviewed by a fresh Codex session (different from the implementation session) that explicitly checks each cited anti-pattern was avoided.

New anti-patterns discovered in production rounds are appended to `vibe-coding-limits-2026.md` with the failing commit sha + remediation sha (success/failure experience encoded).

## §8. Linter Enforcement

- Config: `operator/lint_config.toml`
- Runner: `harness/check_operator_round.py`
- Enforcement owner: cross-validator of the stage, not the performer (performer may run it to catch own errors, but the stage-gate check is on the cross-validator)
- Round 0 scope: **operator docs only** (filename patterns, line caps, required sections, frontmatter validity for meetings, index coverage). Swift linting intentionally out of scope for v5.0; revisit in retro.
- Fail-closed: missing config, missing runner, or non-zero exit = Gate 5 blocker. No "best effort" pass.

## §9. Context Discipline

- **Always-read** per session: persona loader (`.claude/CLAUDE.md` or `AGENTS.md`), `OPERATOR.md`, `FILE_INDEX.md`, active lock if any.
- **On-demand**: use `FILE_INDEX.md` to pick exact files. Targeted `ls`, `rg`, `git diff --name-only` permitted for verification.
- **Forbidden**: undirected bulk-loading (e.g., reading every file in a directory to "build context"), citing legacy docs as authority (see §1), dumping full evaluation reports into meeting bodies (link + hash instead).
- **Unindexed file opens** (opening a file not listed in FILE_INDEX): permitted, but log one line in `context_harness/blackboard.md` with reason. If the file should have been indexed, amend FILE_INDEX.

## §10. SKILLS.md Update Protocol

At round close retro:
1. Classify: did this round reveal a durable pattern not already in SKILLS?
2. If yes → append with S-number, name, pattern, example, why it matters. Mandatory.
3. If no → state explicitly in retro: "no novel pattern identified". Gate 5 advisory, not blocker.

Forbidden: synthesizing a "lesson" from a routine round just to satisfy the rule. Explicit opt-out is preferred to fake content.

## §11. Amendment Protocol for This File

Changes to `REGULATION.md` require:

2. CHANGELOG.md entry with date, summary, meeting pointer
3. Version bump (v5.0 → v5.1 for substantive changes)
4. Operator-layer drift audit re-run to confirm no new contradictions

## Appendix A — Terminology

- **Operator** = one of the two top-level agents (Claude Code, Codex)
- **Performer** = the agent producing an artifact in a given stage
- **Cross-Validator** = the agent reviewing the Performer's artifact in that stage
- **Round** = one execution of plan → design → code → eval → close, bounded by a single contract+lock
- **Stage** = named phase within a round (see `STAGE_CONTRACT.md`)
- **Meeting** = written, markdown exchange between operators with required frontmatter
