## §1. Precedence Ladder

When any two documents conflict, higher precedence wins. An operator may never cite a lower-precedence doc to override a higher one. Citation of a legacy doc to justify a v5 violation = immediate hard stop.

1. **Active round contract + lock** (`operator/contracts/<round_id>/` + `operator/locks/<round_id>.lock`)
2. **`operator/REGULATION.md`** (this file)
3. **`operator/STAGE_CONTRACT.md`**
4. **`operator/OPERATOR.md`**
5. **Domain workflows** (e.g., `docs/design-docs/design-revision-workflow.md`)
6. **Legacy architecture docs** (e.g., `docs/design-docs/multi-agent-architecture.md`) — only if explicitly marked ACTIVE in their own header. Default = SUPERSEDED.

## §3. Contract & Lock

### Per-round directory

```
operator/contracts/<round_id>/
├── spec.md                      ← base input spec. IMMUTABLE after lock.
├── spec.amendment.1.md          ← amendments only as new files
├── spec.amendment.N.md
├── spec.effective.md            ← generated = base + amendments (regenerated by checker)
├── file_whitelist.txt           ← writable paths for this round. IMMUTABLE after lock.
├── file_whitelist.amendment.N.txt  ← whitelist amendments (additive or subtractive)
├── convention_version.txt       ← active coding-conventions path + SHA at lock time
├── lint_config.txt              ← active lint-config path + SHA at lock time
├── acceptance.md                ← success criteria. IMMUTABLE after lock.
├── acceptance.amendment.N.md    ← acceptance amendments
├── eval_protocol.md             ← Codex-authored capture protocol. IMMUTABLE after lock.
└── eval_protocol.amendment.N.md ← eval protocol amendments
```

### Amendment artifact format

All amendments share a common structure:
- Filename: `<base_filename>.amendment.<N>.<ext>` where N is a monotonically increasing integer per target
- First line YAML frontmatter: `target: <base_filename>`, `supersedes: [<section-id or line-range>]`, `meeting: <path-to-meeting>`
- Recorded in lock's `amendments[]` array with `{target, sha256, supersedes, meeting}`
- `spec.effective.md` and equivalent `.effective.*` files regenerated by the checker after every amendment

### Lock file schema

`operator/locks/<round_id>.lock` is a single JSON object:

```json
{
  "round_id": "round_deepsight_r1",
  "schema_version": 1,
  "started_at": "2026-04-19T12:00:00Z",
  "status": "active",
  "operators": {
    "claude_code": { "session": "this-session", "role_default": "implementer" },
    "codex": { "session": "019db43d-...", "role_default": "designer_reviewer" }
  },
  "hashes": {
    "spec.md": "sha256:...",
    "file_whitelist.txt": "sha256:...",
    "convention_version.txt": "sha256:...",
    "lint_config.txt": "sha256:...",
    "acceptance.md": "sha256:...",
    "eval_protocol.md": "sha256:..."
  },
  "amendments": [
    { "file": "spec.amendment.1.md", "sha256": "...", "supersedes": ["spec.md#section-3"], "meeting": "meetings/2026-04-20_amend1.md" }
  ],
  "stages_completed": ["overall_planning", "detailed_design"],
  "closed_at": null
}
```

`status` values: `active` | `paused` | `escalated` | `closed` | `aborted`.

### Immutability rules

- Base contract files (`spec.md`, `file_whitelist.txt`, `convention_version.txt`, `lint_config.txt`, `acceptance.md`, `eval_protocol.md`) are **NEVER edited** after lock creation.
- Direct diff to any base file while `status == active` = hard stop. Checker refuses to advance stages.
- Changes require: (a) amendment meeting → (b) new `<base_filename>.amendment.<N>.<ext>` with frontmatter `target`, `supersedes`, `meeting` → (c) lock's `amendments[]` appended → (d) `<base>.effective.<ext>` regenerated by checker.
- **Whitelist changes** (both expansion AND shrinking) go via `file_whitelist.amendment.N.txt` amendment. No exception. "Shrink without meeting" is forbidden — mutating the locked hash, regardless of direction, breaks the immutability invariant.
- `status` transitions: `close <round>` subcommand of `harness/check_operator_round.py` transitions to `closed` only if all Gate 1–5 pass. Operators may transition to `paused` or `escalated` via meeting; `aborted` only via human. Operators may NOT hand-edit `status` in the lock file.
- `lint_config.txt` and `convention_version.txt` contain `<path>\n<sha256>`. Checker at Gate 5 recomputes the live SHA of the referenced path and compares — mismatch = blocker.

## §4. Challenge Section (anti-hallucination tripwire)

Every meeting whose purpose is a **decision** (normative, design-affecting, or contract-changing) must include a Challenge Section with at least ONE of:

- Concrete objection with reasoning
- Named risk with mitigation or acceptance
- Rejected alternative with why-not
- Explicit uncertainty that was consciously deferred

Meetings that are **purely factual** (verification, status check, evidence pointer) must state: **"No normative decision; verified facts only"** and list the evidence references.

Forbidden: fake dissent (reviewer nitpicking to satisfy the rule). If the reviewer genuinely has no objection, the meeting is factual by definition — state that.

Missing Challenge Section on a decision meeting = Gate 5 blocker.

## §5. Gate Structure

Gates 1–4 from v4 retained with enforcement clarified; Gate 5 added.

| Gate | Name | Check | Blocker |
|------|------|-------|---------|
| 1 | Build & Test | `xcodegen generate && xcodebuild test` passes, test count ≥ baseline | Any failure |
| 2 | 3-Evaluator Review | red_team + hig_guardian + visual_qa reports produced | Any evaluator missing or empty |
| 3 | Cross-Agreement | 2+ evaluators agreeing on a file → required fix | Unresolved 2+ agreement |
| 4 | Process Metrics | remediation cycles ≤ 2, BLOCKER recurrence = 0, brief accuracy ≥ 90%, SKILLS miss ≤ 5% | Any metric out of range |
| 5 | Process Integrity | see §5.1 below | see §5.1 blockers |

### §5.1 Gate 5 — Process Integrity

Blockers (round cannot close):
- Any commit in this round not traceable to a contract lock
- Any edit to a base contract file (§3 immutability)
- Any decision meeting without a Challenge Section
- Any new operator/* file not registered in `FILE_INDEX.md`
- Lock hash mismatch for any base contract file
- Lint checker missing, failing, or unrunnable (`harness/check_operator_round.py` exit != 0)
- Operator-layer drift audit reports contradictions (see §7)

Advisories (noted in retro, do not block):
- No new SKILLS.md entry — acceptable if retro explicitly classifies "no novel pattern"
- FILE_INDEX line-count warning (between 250 and 300 lines)
- SESSION_RESUME.md staleness > 48h

## §6. Deadlock Escalation Ladder

When two operators cannot converge in a meeting:

1. **Reframe meeting** — each operator restates the OTHER'S position in their own words. Names the real decision axis. Frequently resolves miscommunication-based disputes.
2. **Default-reversible rule** — if the dispute is low-impact and reversible, pick the option with smaller blast radius, easier rollback, or better auditability. Record both positions; implementer proceeds. No human, no Gemini.
3. **Gemini advisory** — for **bounded technical disputes only** (lint rule choice, file layout, test strategy, refactor approach). Gemini provides ANALYSIS, not VERDICT. Claude/Codex still make the decision and record whether they accepted or rejected the advice.
4. **Human escalation** — required for: product-impact, user-visible design direction, security/privacy policy, architecture change, irreversible migration, or unresolved operator deadlock after Gemini advisory.

Timebox: 2 written counter-rounds OR 24 hours — whichever first — before moving to the next tier.

Lock `status` transitions to `escalated` at Tier 4. Must record: both positions, tier path taken, what was asked of the human.

## §7. Operator-Layer Drift Audit

Runs at Gate 5 and after every operator-doc amendment. Implemented in `harness/check_operator_round.py` (section `audit_operator_layer`).

Checks for:
1. Role matrix consistency across OPERATOR/STAGE_CONTRACT/REGULATION — same agent, same stage, same role everywhere
2. Stage names identical across all operator docs
3. Every file mentioned in operator/* exists
4. Every operator/*.md file (except CHANGELOG and meetings/*) is indexed in FILE_INDEX.md
5. Line-count caps honored: AGENTS.md ≤80, .claude/CLAUDE.md ≤80, OPERATOR.md ≤120, FILE_INDEX.md ≤250
6. No legacy doc cited at higher precedence than a v5 operator doc
7. SESSION_RESUME.md last-updated timestamp < 48h (warn) / < 7d (block)
8. `status: active` lock implies every stage_completed in the matrix was run through the expected performer + cross-validator pair

Any contradiction → Gate 5 blocker. Resolution path: amendment meeting.

## §8. Linter Enforcement

- Config: `operator/lint_config.toml`
- Runner: `harness/check_operator_round.py`
- Enforcement owner: cross-validator of the stage, not the performer (performer may run it to catch own errors, but the stage-gate check is on the cross-validator)
- Round 0 scope: **operator docs only** (filename patterns, line caps, required sections, frontmatter validity for meetings, index coverage). Swift linting intentionally out of scope for v5.0; revisit in retro.
- Fail-closed: missing config, missing runner, or non-zero exit = Gate 5 blocker. No "best effort" pass.

## §9. Context Discipline

- **Always-read** per session: persona loader (`.claude/CLAUDE.md` or `AGENTS.md`), `OPERATOR.md`, `FILE_INDEX.md`, active lock if any.
- **On-demand**: use `FILE_INDEX.md` to pick exact files. Targeted `ls`, `rg`, `git diff --name-only` permitted for verification.
- **Forbidden**: undirected bulk-loading (e.g., reading every file in a directory to "build context"), citing legacy docs as authority (see §1), dumping full evaluation reports into meeting bodies (link + hash instead).
- **Unindexed file opens** (opening a file not listed in FILE_INDEX): permitted, but log one line in `context_harness/blackboard.md` with reason. If the file should have been indexed, amend FILE_INDEX.

## §10. SKILLS.md Update Protocol

At round close retro:
1. Classify: did this round reveal a durable pattern not already in SKILLS?
2. If yes → append with S-number, name, pattern, example, why it matters. Mandatory.
3. If no → state explicitly in retro: "no novel pattern identified". Gate 5 advisory, not blocker.

Forbidden: synthesizing a "lesson" from a routine round just to satisfy the rule. Explicit opt-out is preferred to fake content.

## §11. Amendment Protocol for This File

Changes to `REGULATION.md` require:
1. A meeting with `status: decided` and a Challenge Section (§4)
2. CHANGELOG.md entry with date, summary, meeting pointer
3. Version bump (v5.0 → v5.1 for substantive changes)
4. Operator-layer drift audit re-run to confirm no new contradictions

## Appendix A — Terminology

- **Operator** = one of the two top-level agents (Claude Code, Codex)
- **Performer** = the agent producing an artifact in a given stage
- **Cross-Validator** = the agent reviewing the Performer's artifact in that stage
- **Round** = one execution of plan → design → code → eval → close, bounded by a single contract+lock
- **Stage** = named phase within a round (see `STAGE_CONTRACT.md`)
- **Meeting** = written, markdown exchange between operators with required frontmatter
