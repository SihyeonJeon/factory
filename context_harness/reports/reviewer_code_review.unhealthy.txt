## CRITICAL (Must fix before release)

### C1. Open redirect in OAuth callback
**File:** `src/app/auth/callback/route.ts:18,26`

```ts
const next = searchParams.get("next") ?? "/";
const response = NextResponse.redirect(`${origin}${next}`);
```

The `next` parameter is user-controlled and not validated. An attacker can craft a callback URL like `/auth/callback?code=...&next=//evil.com` which would redirect to `https://evil.com` (protocol-relative URL) or `next=@evil.com` depending on URL parsing. This is a classic open redirect vulnerability.

**Fix:** Validate that `next` starts with `/` and does not start with `//`:
```ts
const rawNext = searchParams.get("next") ?? "/";
const next = rawNext.startsWith("/") && !rawNext.startsWith("//") ? rawNext : "/";
```

### C2. Event page and dashboard use mock data, not Supabase
**File:** `src/app/event/[id]/page.tsx:12-13,42-43` and `src/app/dashboard/[id]/page.tsx:18-19`

Both the event page and dashboard page call `getMockEvent(id)` which returns hardcoded data regardless of the ID:

```ts
const event = getMockEvent(id);  // Always returns "금요 와인 모임"
```

This means:
- Every event URL shows the same event
- RSVP submissions go to `console.log` only (`event-rsvp-flow.tsx:46`)
- Dashboard shows mock data, not real guests (though `useRealtimeGuests` fetches real data, the event metadata is mock)
- Event creation uses `alert()` instead of persisting (`create-event-wizard.tsx:76-78`)

The photos page (`src/app/event/[id]/photos/page.tsx:12-49`) is the only page that correctly falls back to mock only when Supabase returns null. **The event page and dashboard must follow the same pattern.**

### C3. `fcm_token` column exposed through public profiles RLS
**File:** `supabase/migrations/00002_rls_policies.sql:19-21`

```sql
create policy "profiles: public read"
  on public.profiles for select
  using (true);
```

The profiles table now has `fcm_token` (added in migration 00004), and the RLS policy allows **anyone** to read **all columns** of all profiles. This leaks FCM push tokens to any authenticated or anonymous user, enabling push notification spam.

**Fix:** Either restrict the public read policy to specific columns using a view, or add a separate restrictive policy and make the base policy more selective. Simplest approach: create a `profiles_public` view with only `id, display_name, avatar_url` and use that in queries.

## HIGH (Should fix before release)

### H1. Service worker caches API responses including auth data
**File:** `public/sw.js:39-49`

The service worker uses network-first caching for `/api/` routes, meaning API responses (potentially containing user-specific data, auth tokens, or error messages with sensitive info) are cached in the browser's CacheStorage. This is risky on shared devices.

**Fix:** Exclude `/api/` routes from SW caching entirely, or limit caching to specific safe endpoints.

### H2. No CSRF protection on API routes
**Files:** `src/app/api/media/upload/route.ts`, `src/app/api/fcm-token/route.ts`, `src/app/api/reminders/send/route.ts`

All three API routes accept POST requests authenticated only by Supabase session cookies. Since cookies are automatically sent by the browser, a malicious site could forge requests (CSRF). While Supabase's `getUser()` call validates the session, it doesn't prevent CSRF — the cookies will be attached automatically.

**Fix:** Consider checking the `Origin` or `Referer` header, or requiring a custom header (e.g., `X-Requested-With`) that browsers won't send in cross-origin simple requests.

### H3. `settlement-calc` edge function uses `service_role` key to create the Supabase client but then calls `auth.getUser(token)` to verify the caller
**File:** `supabase/functions/settlement-calc/index.ts:84-85,93-101`

```ts
const supabase = createClient(supabaseUrl, serviceRoleKey);
// ...
const { data: { user } } = await supabase.auth.getUser(token);
```

This pattern is correct for verifying the caller, but all subsequent DB operations bypass RLS (since the client uses `service_role`). The `mark_paid` action at line 213 does verify host ownership, which is good — but any bug in the authorization checks would be unmitigated by RLS. Document this as an intentional design choice and ensure all action paths have explicit authorization checks.

### H4. No rate limiting on photo upload
**File:** `src/app/api/media/upload/route.ts`

The per-event limit of 10 photos is checked, but there's no per-user rate limit. An authenticated user could spam upload requests, consuming storage and bandwidth. Consider adding a simple in-memory or Redis-based rate limiter.

## MEDIUM (Should fix, not blocking)

### M1. `send-reminder` edge function uses non-URL-safe base64 for JWT
**File:** `supabase/functions/send-reminder/index.ts:35-36,68-71`

The JWT construction uses `btoa()` which produces standard base64, not base64url. Google's OAuth2 endpoint expects base64url encoding (no `+`, `/`, or `=` padding). This will fail at runtime.

**Fix:** Use a `btoa`-to-base64url conversion:
```ts
function base64url(str: string): string {
  return btoa(str).replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
}
```

### M2. Signed URLs expire but no refresh mechanism
**File:** `src/lib/queries/media.ts:44` and `src/lib/storage.ts:62`

Photos page creates 24-hour signed URLs on server render. For users who keep the page open, images will break after 24 hours. The upload endpoint creates 7-day URLs, but these are also one-shot.

**Recommendation:** Add a client-side mechanism to detect expired URLs and re-fetch, or use longer expiry times.

### M3. `theme_color` mismatch between `manifest.json` and `layout.tsx`
**Files:** `public/manifest.json:8` (`#FF6B9D`) vs `src/app/layout.tsx:32` (`#ffffff`)

The PWA manifest declares `theme_color: "#FF6B9D"` (pink) but the viewport meta declares `themeColor: "#ffffff"` (white). This creates inconsistent status bar colors between installed PWA and browser view.

### M4. `ExternalBrowserBanner` imports `lucide-react` but no other component does
**File:** `src/components/kakao/external-browser-banner.tsx:8`

```ts
import { X } from "lucide-react";
```

Confirm `lucide-react` is in `package.json`. If it's only used for this one icon, consider using an inline SVG to reduce bundle size (consistent with the pattern used everywhere else in the codebase).

### M5. `companion_count` not validated for upper bound on client
**File:** `src/components/rsvp/rsvp-details-form.tsx` (not read, but the DB constraint is `check (companion_count >= 0 and companion_count <= 10)`)

The DB enforces 0-10 range, but the client should also enforce this to provide a good UX and avoid silent rejections.

### M6. Multiple Supabase clients created per render in `useRealtimeGuests`
**File:** `src/hooks/use-realtime-guests.ts:58,70`

`createClient()` is called both inside `fetchProfile` (via useCallback) and inside the useEffect. The `@supabase/ssr` `createBrowserClient` returns a singleton, so this works, but it's better practice to create it once at the top of the effect.

## LOW (Nice to have)

### L1. PWA manifest uses SVG icons only
**File:** `public/manifest.json:12-29`

Some older Android browsers and some PWA install prompts require raster PNG icons (at least 192x192 and 512x512). SVG-only manifests may prevent install on some devices.

### L2. No `<noscript>` fallback
**File:** `src/app/layout.tsx`

The entire app is client-rendered React. No `<noscript>` tag for users with JS disabled (rare but good practice for SSR).

### L3. Service worker `skipWaiting()` without user prompt
**File:** `public/sw.js:15-16`

`skipWaiting()` immediately activates the new service worker without asking the user. For a PWA, this can cause page reloads mid-interaction. Consider a "New version available" prompt pattern.

### L4. OG image generation fetches from Supabase but event page uses mock data
**File:** `src/app/event/[id]/opengraph-image.tsx:22` vs `src/app/event/[id]/page.tsx:12`

The OG image route correctly queries Supabase, while the page itself returns mock data. This means the OG preview shows correct data but the actual page shows hardcoded "금요 와인 모임" — confusing for users clicking through from KakaoTalk shares.
